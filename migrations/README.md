# Migrations

Plain SQL migrations applied by `scripts/migrate-safe.sh` in lexical order.
Each file's basename is its version key in `schema_migrations`.

## Naming

```
NNNN_short_description.sql
```

- `NNNN` is a zero-padded four-digit sequence.
- `short_description` is snake_case.
- One logical change per file.

## Rules

- Run real database migrations only through `just db-migrate-dev` or
  `just db-migrate-deploy`; both call `scripts/migrate-safe.sh` and take a
  backup first.
- Migration SQL runs through `psql --single-transaction`. DDL that requires its
  own transaction, such as `CREATE INDEX CONCURRENTLY`, needs a separate
  operator path before it can be introduced.
- The migration set starts from a canonical schema baseline. After the initial
  public baseline is cut and a migration is applied to a persistent database,
  add a new migration instead of editing history.
- Numbering after the baseline starts with `0003_*`.
- Adding a source after the baseline: append it to the registry so existing
  source ids stay stable, regenerate the `0002` fresh-database seed with
  `just gen-source-artifacts`, and add a forward migration that inserts the new
  row idempotently for databases that already applied the previous `0002`. The
  regenerated `0002` covers fresh databases; the forward migration covers the
  existing one.
- Materialized view refresh strategies should be documented in the migration
  that creates the view and exercised by a test.

## Current Migrations

The live on-disk migration set is a canonical baseline and the generated source
seed. Future changes are appended as forward migrations.

| File | Notes |
|------|-------|
| `0001_canonical_schema.sql` | Canonical schema baseline. Creates producer base tables, chain sidecars, poll cursors (incl. `target_height`), read-model tables, source-health invariants (incl. strict/weak orphan counters), Bitcoin Core backbone state, the complete-canonical tree index, event-level pool attribution, the Hathor reward-output columns, and the `rsk_reclassify_watermark` singleton directly. |
| `0002_seed_sources.sql` | Registry-generated fresh-database seed for every `source` row from `crates/mmm-capture/src/source_registry` in fixed id order (the 34 live, historical, and catalogued sources). It raises if `source` is already populated. Generated by `just gen-source-artifacts`; do not edit by hand. |

`0002_seed_sources.sql` is only for fresh/reset databases. It intentionally
raises if `source` is already populated, so a non-reset database fails cleanly
after the wrapper backup instead of appending rows with wrong ids.

If any migration fails, both dev and deploy mode restore from the just-created
backup and exit non-zero.

Manual restore uses the same compatibility filter printed by the wrapper:

```sh
gunzip -c '<backup-file>' | sed '/^SET transaction_timeout/d' | psql --set ON_ERROR_STOP=on
```

## Reset Path

Fresh, CI, and throwaway databases use the explicit reset path. `just clean`
does not remove the Postgres volume.

1. Confirm the database is disposable local state.
2. `CONFIRM_DB_RESET=1 just db-reset` (runs `just db-backup` first).
3. `just db-up`
4. `just db-migrate-dev`
5. `just rebuild-source-health`
6. Re-run required historical backfills or live-test backfill commands.

## Deferred Objects

Some planned objects are deliberately absent from the current schema. They
should land together with their owning consumers:

| Object | Lands with |
|--------|------------|
| `tip_state` and full live-chaintip observation semantics | the Bitcoin Core live-chaintip producer |
| `reconcile_queue` | queue-based reconcile workers |
| `v_timeline_daily`, `v_pool_period_stats`, `v_near_miss_rollup` | API, UI, or analytics consumers |

## Postgres Type Choices

| Concept | Postgres representation | Notes |
|---------|----------------|-------|
| Surrogate keys | `BIGINT GENERATED ALWAYS AS IDENTITY` | SQL-standard identity form. |
| Epoch timestamps | `BIGINT` | Avoids 2038 wraparound. |
| Validation flags | `BOOLEAN` | |
| Hashes, headers, and proofs | `BYTEA` | Header hashes, coinbases, and merkle proofs. |
| Structured metadata | `JSONB` | Pool tags and payout addresses. |

`CHECK` constraints encode source vocabulary, parent-kind vocabulary, chain
sidecar invariants, byte-length invariants for hashes and headers, and
read-model consistency constraints. Producer write SQL lives in
`crates/mmm-store/src/lib.rs`; derived-state writes go through
`crates/mmm-read-model`.
